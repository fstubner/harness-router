# v0.3 Spec — harness-router

**Goal.** Cut surface area roughly in half. Make the config match how users
think (model-first, not service-first). Stop maintaining anything that
depends on a hardcoded model catalog. Share quota state across multiple
stdio servers so concurrent agents don't fight over the same rate limit.

**Greenfield.** v0.2.0 is published but the user base is small enough that
we cut over hard. No deprecation aliases, no dual npm publish, no rc cycle.
A migrator exists for v0.2.0 users; v0.2.1 is folded into v0.3 unshipped.

---

## Naming & invocation

- npm package: `harness-router-mcp` → `harness-router`. Hard cutover.
- Local folder: `coding-agent-mcp` → `harness-router`. (Out-of-tree
  rename; user does this when convenient.)
- Bare `npx harness-router` runs the stdio MCP server. The `mcp`
  subcommand goes away.
- Subcommands: `onboard`, `doctor`, `dashboard`, `install`, `uninstall`,
  `migrate` (new), `auth` (new), `serve` (HTTP, was `http`).
- Internal names ending in `coding-agent-mcp` get renamed too (binary,
  MCP server name, otel service name, etc.) — single audit pass.

## Tool surface

- One tool: `code`. Inputs: `prompt`, optional `files`, `workingDir`,
  `hints`, `mode: "single" | "fanout"`, `models` (only when `mode=fanout`).
- One resource: `harness-router://status` — plain-text dashboard, fetched
  on demand.
- No prompts.
- Deleted: `code_mixture` tool, `dashboard` tool, `get_quota_status` tool,
  `route-task`/`compare-models`/`health-check` prompts.

## Config schema

- Top-level keyed by model. Each entry has `subscription?`, `metered?`, or
  both.
- `tier` field gone — derived from which key the route lives under.
- `cli_model` gone. Users write pinned IDs everywhere
  (`claude-opus-4-7`, not `opus`). Optional `cli_model_override` is the
  escape hatch when canonical ≠ CLI flag.
- `priority` and `mixture_default` reference model keys (no service names).
- `http` block carries optional auth + bind/port.
- Path stays `~/.harness-router/config.yaml`.

## Shared quota state

- Replace per-PID delta files with a single SQLite database at
  `~/.harness-router/state.db`.
- Schema: `quota_counters(service, window_start, count, …)`,
  `breaker_state(service, tripped_until, failure_count, …)`,
  `last_headers(service, timestamp, raw_json)`.
- Every stdio/HTTP server opens the same DB in WAL mode. Concurrent
  reads/writes handled by SQLite. No daemon, no IPC, no lifecycle.
- `better-sqlite3` as the dep. Prebuilt binaries for macOS / Linux /
  Windows on Node ≥ 20.
- Migration on first run: if old `quota_state.json` or
  `quota_state/{pid}-*.json` files exist, fold their counts into the
  fresh DB and delete the legacy files.

## Routing algorithm

Unchanged. Walk priority; per model, subscription before metered;
quota score tiebreak; circuit-breaker on failures.

## Catalog

- No hardcoded list anywhere in the code.
- Wizard hits OpenRouter `https://openrouter.ai/api/v1/models` once at
  onboard time, filters to Anthropic / OpenAI / Google, presents as a
  checkbox.
- On any failure (network, 4xx, malformed), falls through to free-text
  input. Free-text is always available alongside the picker for local
  models / niche providers.

## HTTP transport

- Default bind: `127.0.0.1:8765`. Auth optional and off by default for
  loopback.
- Non-loopback bind force-enables auth.
- Token at `~/.harness-router/auth.token` chmod 600.
- `harness-router auth { create | show | rotate | revoke }`.

## Migration

- v0.3 loader detects v0.2 shape (`services:` or `overrides:` at top
  level, no `models:`) and refuses to run with: "run `harness-router
  migrate`."
- `harness-router migrate` reads v0.2 YAML, writes v0.3 YAML, prints
  warnings (services dropped, duplicates collapsed, mixture_default
  service-names → model-names).
- Backup: `config.yaml` → `config.yaml.v2.bak` before overwrite.
- Quota state migrated automatically on first server start (see Shared
  quota state above).

## Source-tree restructure

`src/` flattens become explicit boundaries (PORTFOLIO risk):

```
src/
  config/        — V3Config types, loader, migrator, openrouter client
  routing/       — router, circuit-breaker
  state/         — SQLite quota + breaker store
  dispatchers/   — unchanged file-by-file, just a clearer parent dir
  mcp/           — server, tools (one tool), resources (one resource)
  cli/           — bin entrypoint, subcommand modules
  install/       — host adapters
  observability/ — unchanged
```

## Out of scope for v0.3

- Doctor live-dispatch probe (still planned but ships later).
- MCP resource subscriptions / live-updating dashboard.

## Open questions (yes/no each)

1. **`cli_model_override`** — keep as undocumented escape hatch, or drop
   entirely and require pinned IDs always?
   _Proposed: keep, undocumented._
2. **Folder rename** (out-of-tree) — do it now or defer?
   _Proposed: defer to a one-shot `mv` whenever convenient. Doesn't
   block the npm/binary rename._
3. **SQLite dep on Windows** — `better-sqlite3` has Windows prebuilds
   but installation occasionally requires VS build tools when prebuilds
   miss. Acceptable, or pure-JS fallback (`sql.js`) for Windows
   reliability?
   _Proposed: better-sqlite3 with no fallback. If Windows install
   breaks for a user, they file an issue._
